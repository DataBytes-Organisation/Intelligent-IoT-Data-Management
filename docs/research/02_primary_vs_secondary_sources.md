## What this resource contains

- Definitions and examples of primary vs secondary sources across disciplines.
- Guidance on selecting source types based on assessment requirements.
- Clarification that source type can vary by context and intended use.

## Primary source selection criteria for this project

- Prefer original technical artifacts as primary evidence: source code, test outputs, API responses, commit history, meeting notes.
- Use generated documents (`HLD.md`, `LLD.md`) as project-internal primary evidence of design intent.

## Secondary source usage boundaries

- Use external articles/guides for framing and method support only.
- Do not treat third-party blog claims as implementation proof without local artifact evidence.

## How source quality influenced our technical decisions

- Strengthened evidence strategy by linking every claim to a repository path or command output.
- Prioritized reproducible outputs (tests, endpoint checks) over descriptive statements.
